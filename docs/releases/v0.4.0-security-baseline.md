## Features

- Adiciona uma barreira HTTP Basic stateless às APIs de negócio de Catalog,
  Inventory e Order.
- Exige uma única credencial técnica por ambiente para todas as rotas
  `/api/v1/**`.
- Mantém públicas apenas as exceções técnicas `GET /actuator/health`, OpenAPI e
  Swagger UI; elas não concedem acesso às operações de negócio.
- Faz o Order autenticar as chamadas REST existentes ao Inventory com a
  credencial configurada localmente, sem propagar a credencial recebida do
  consumidor.

## Improvements

- As coleções Postman agora expõem variáveis vazias e não versionadas para
  `apiUsername` e `apiPassword`.
- A documentação de execução dos três serviços declara as variáveis obrigatórias
  `SECURITY_API_USERNAME` e `SECURITY_API_PASSWORD` e o limite de uso de HTTP
  Basic fora do ambiente local.

## Quality validation

- As suítes automatizadas de Catalog, Inventory e Order passaram sem falhas,
  erros ou skips; a suíte Inventory com profile Maven `kafka` também passou.
- Chamadas sem credencial às rotas representativas de negócio retornaram `401`;
  a escrita não autenticada no Inventory foi recusada antes de criar estado.
- Chamadas autenticadas preservaram criação e consulta de recursos e o fluxo
  Order → Inventory.
- `E2E-046-001` confirmou startup do Inventory com profile Spring `kafka`,
  publicação de `OrderConfirmed` v1, consumo pelo Inventory e observação com
  estado `COMPLETED`.

A evidência detalhada está no [Test Report da Story #46](../quality/story-046/TEST_REPORT.md).

## Breaking changes

Consumidores das APIs de negócio devem enviar `Authorization: Basic ...` com
credenciais válidas. Uma solicitação ausente, inválida ou malformada recebe
`401 Unauthorized` com `WWW-Authenticate` antes de alcançar regra de negócio.

## Migration notes

1. Forneça `SECURITY_API_USERNAME` e `SECURITY_API_PASSWORD` somente no
   ambiente de execução; não versione, registre em logs ou inclua seus valores
   em exemplos e evidências.
2. Configure o cliente de API para enviar HTTP Basic a todas as chamadas de
   negócio em `/api/v1/**`.
3. Para execução remota, disponibilize HTTP Basic exclusivamente sobre HTTPS
   confiável. TLS, gateway, rotação de credenciais, tokens e gestão de
   identidade não fazem parte desta Story.
4. A integração interna Order → Inventory usa as mesmas variáveis locais; não
   há endpoint interno alternativo nem bypass de autenticação.

## Known limitations

- Há somente uma identidade técnica compartilhada por ambiente: não há
  usuários, roles, autorização por recurso, auditoria de acesso ou revogação
  granular.
- `OrderConfirmed` v1, seu tópico, envelope, producer, consumer e grupo Kafka
  permanecem inalterados.
